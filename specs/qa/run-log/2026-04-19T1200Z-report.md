## 2. Achados por Cenário

### 2.1 stat-01 — Duplicação de auth fora de `core.js`

**Status:** FAIL
**Prioridade:** P2 (manutenibilidade / risco de inconsistência de sessão)
**Tipo:** static

**Hipótese do cenário:** Chamadas a `signInWithPassword`, `signUp`, `signOut` e helpers de sessão Supabase devem ser centralizadas em `js/core.js`. Páginas fora de `core.js` devem consumir apenas os helpers exportados.

**Evidências coletadas (grep):**

- **login.html:601** — `await _sb.auth.signInWithPassword({ email, password });`
  → **LEGÍTIMO.** `login.html` é o ponto canônico de login; espera-se `signInWithPassword` inline nesta página. Alinha com `autenticacao.spec.md`.

- **dashboard-detentor.html:1992, 2002, 2506, 4344, 4391, 4505, 4544** — múltiplas chamadas a `sb.auth.setSession(...)` e `sb.auth.getSession()`.
- **dashboard-marca.html:1725, 3546, 3585, 3707, 3742, 3941, 3971, 4251, 4322** — mesma coisa, 9+ call sites.
- **criar-oportunidade.html:3737, 3746, 8429, 8475** — `sb.auth.setSession`, `getSession`, `getUser`, `signOut`.
- **prototipo-live-preview.html:1835, 1844** — `sb.auth.setSession` + `getSession` (arquivo de preview, não documentado em surface-map — ver nota spec-drift abaixo).
- **criar-oportunidade.html.bak:5896, 5901, 5906, 5916, 5942** — arquivo `.bak` residual com mesmas chamadas.

**Padrão observado em linha 3730-3748 de `criar-oportunidade.html` (trecho representativo):**

```
var session = null;
if (typeof sb !== 'undefined' && sb) {
  var stored = localStorage.getItem('sb_detentor_session');
  if (stored) {
    try {
      var detentorSession = JSON.parse(stored);
      if (detentorSession.access_token && detentorSession.access_token !== 'DEMO_TOKEN') {
        await sb.auth.setSession({ access_token: ..., refresh_token: ... });
      }
    } catch (_) {}
  }
  try {
    var data = await sb.auth.getSession();
    ...
  } catch (e) { ... }
}
```

Este bloco de "restore SDK a partir de localStorage → depois getSession" é **exatamente** o que o helper `restoreSessionToSDK()` em `js/core.js` foi projetado para fazer. O fato de estar reimplementado inline 20+ vezes significa:

1. Qualquer futura mudança na lógica de refresh/cascade de sessão precisa ser propagada manualmente a cada call site — risco alto de drift.
2. O helper `getValidToken()` em `core.js` já encapsula a cascata "SDK → localStorage → refresh com AbortController 5s". As páginas não o usam de forma consistente — `dashboard-marca.html:3966-3968` usa `getValidToken`, mas logo abaixo (3969-3974) refaz `sb.auth.getSession()` manualmente como fallback.
3. Falhas silenciosas (`catch (_) {}`) em 10+ locais mascaram problemas de refresh de token.

**Veredicto:** FAIL — duplicação extensa e real. Não há credenciais vazadas nem vulnerabilidade imediata, mas há **dívida técnica significativa** e risco de drift.

**Referência cruzada:**
- Surface-map seção 3.1 descreve o "Dual Auth Pattern" como padrão aceito, mas deixa explícito que `core.js` é a fonte de verdade dos helpers — o desacoplamento atual vai contra isso.
- `criar-oportunidade.html.bak` não é apenas duplicação de auth, mas **arquivo backup residual em produção**, que deve ser removido ou movido para fora de `/MVP/` (ver próximos passos).

**Veredicto:** FAIL — a Edge Function `invite-org-member` é invocada em produção por dois dashboards principais (marca e detentor) sem qualquer documentação em `surface-map.md`. Isso significa:

1. O comportamento, autenticação e RLS da função não foram auditados pela Fase 2 (security) nem pela Fase 3 (persona), pois essas fases usam `surface-map.md` como contrato.
2. Não há clareza sobre qual role (anon vs. service_role) a função usa internamente, nem quais tabelas ela toca.
3. Potencial ponto cego para vulnerabilidades tipo "confused deputy" (função com service_role expondo operações para usuários autenticados sem validação de autorização no corpo).

**Nota adicional:** `create-user-and-send-access` está parcialmente documentada (uma menção em prosa, sem seção 8.x estruturada) — recomenda-se promovê-la a entrada de seção 8 também.
